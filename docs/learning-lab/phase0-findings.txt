RISKS:
  - Security regression in flight: the working branch feat/goodnbad-os-secure-slice carries the OLD lead-based entitlement (lib/vault/entitlement.ts:45-51 — any visitor who submits the free /subscribe form with a track becomes 'entitled'), while origin/main already shipped the hardened sales-gated version (commit 7b094eb, PR #48).   - No user identity layer anywhere: entitlement is keyed purely on a client-asserted email (POST /api/vault/sign accepts any {email} with no ownership proof), and there are no accounts, sessions, or magic links. A 13-state Learning Lab lifecycle cannot safely hang per-user state off an unverified email — an auth/verification layer is a prerequisite, not a reuse.
  - The flags/next mechanism is unproven for dark-shipping: the single existing flag (facebookSDK) is a secret-holder, not a boolean UI gate; whether the FLAGS/FLAGS_SECRET Vercel integration is actually configured in the production project could not be verified locally (runtime Vercel env is unreadable from this workspace).

  - Lifecycle state today is implicit and scattered: lead → sale → drip-week-N → refunded is derived at read time across three tables (leads, sales, drip_sends) on THREE different branches (secure-slice, main, feat/weekly-drip + integration/phase-1-2, both unmerged).   - Watermark cache (lib/vault/watermark.ts) and updates store are in-memory per lambda instance — fine for PDFs, but the same pattern would silently lose Learning Lab progress state if copied; per-user lifecycle state must live in Supabase.
  - middleware.ts dark-shipping returns 404 by prefix only — anything NOT listed in both PRODUCTION_BLOCKED_PREFIXES and the matcher ships publicly by default; a Learning Lab route added under an unlisted path is live immediately on merge because Vercel auto-deploys main (repo CLAUDE.md 'auto-syncs with Vercel on push').

RECOMMENDATION: Treat the vault loop as the reference architecture but reuse selectively. (a) Reuse directly: the Supabase service-role pattern (server-only client, RLS-on/no-anon tables), the zod-at-the-boundary API style, the manifest-allowlist + HMAC short-TTL signed-URL + watermark delivery chain (lib/vault/sign.ts is generic and can sign any Learning Lab claim), the teaser-vs-full free/paid split, and the WeekModule dependsOn curriculum type in lib/subscribe/tracks.ts as the seed for lesson prerequisites. (b) Base all paid-state logic on origin/main's entitlement.ts (hasConfirmedSale against `sales`), NOT the working branch's lead-based version — flag the divergence to whoever merges feat/goodnbad-os-secure-slice. (c) For the 13-state lifecycle, do not force-fit the current implicit lead/sale/drip states: add one new RLS-locked Supabase table (e.g. learning_state: email/user_id, state enum, per-lesson progress jsonb, timestamps) written only by service-role routes, and adopt the drip_sends claim-before-act idempotency pattern from feat/weekly-drip for any state transition with side effects. (d) Ship dark using both existing mechanisms: put the Learning Lab under a dedicated prefix added to middleware PRODUCTION_BLOCKED_PREFIXES + matcher while unfinished, and introduce the first real boolean flag in flags/index.ts (decide() reading an env var works today even without the Vercel FLAGS integration) to flip it live without a route change. (e) Resolve the identity gap first — email-only entitlement is acceptable for one-shot PDF downloads but not for a stateful lifecycle; plan at minimum a signed magic-link email verification (SMTP/Resend plumbing already exists in both the lead route and lib/email/resend.ts). (f) Before building, confirm on Vercel that VAULT_SIGNING_SECRET, SUPABASE_*, and the flags integration are actually configured in production — this could not be verified from the local workspace.
