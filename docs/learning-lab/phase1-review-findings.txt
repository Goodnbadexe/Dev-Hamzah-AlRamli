LENS: Scope guardian — MVP boundary enforcement (USAGE-RULES.md "Do not build a giant autonomous Learning OS yet" + NEXALIS minimal-build policy)

VERDICT: The Phase 1 set is more disciplined than most drafts of this kind — the 13-state enum matches USAGE-RULES verbatim, payment processing, AI plan generation, admin dashboard UI, and pricing are all correctly deferred, and reuse of the repo's zod/SMTP/flag/middleware patterns is deliberate rather than duplicative. But scope creep concentrates in three subsystems that the MVP boundary ("email notifications", "basic checks", one commercial loop) does not license: an 18-event notification matrix whose cron-driven half (deadline sweeps, nudge counters, digests, retry sweeper, T-1h reminders) is both unnecessary for a one-person service and undeliverable on the verified single daily Vercel cron; an anonymous signed-upload pipeline plus a learning_files registry that duplicates Supabase Storage metadata and feeds a scan_status no scanner exists to fill; and retention/archival automation misjustified by a NEXALIS rule that governs the reliability monitor, not product data. Cutting to ~10 synchronous notification events, text-only intake with post-auth uploads, seq-only module ordering, and manual retention removes roughly a third of Phase 3 surface without touching a single USAGE-RULES requirement. Two genuine gaps need fixing before build, though: the loop has no specced learner surface for accepting a quote, replying to clarification, or submitting payment evidence (portal access only begins after payment — the first learner stalls at the quote email), and the blessed interim Supabase-dashboard admin surface would bypass the audit-trail invariant and the step-release email unless it is restricted to content drafting while the server-action set gains release-step and save-feedback verbs.

LENS: Security — plan-level design review of the Learning Lab Phase 1 spec set (identity/magic-link flow, owner gate, payment-evidence lifecycle, uploads, rate limiting, notifications, policy backing)

VERDICT: The spec set is unusually disciplined for a Phase 1 draft — the 13-state lifecycle with owner-only payment verification, append-only audit trail, service-role-only data access, and 'never trust a learner claim' rule are the right skeleton, and PHASE0-EVIDENCE grounds it honestly in what the codebase actually has. But the design has one structural fault line and several unresolved seams on exactly the surfaces that carry money and PII. The fault line is identity timing: the docs contradict each other on WHEN the learner becomes authenticated (DATA-MODEL §4 says accept-quote/submit-evidence are authenticated learner actions; PRD §6 and NOTIFICATIONS row 7 grant portal access only after payment) — as written, the commercially binding pre-payment actions ride on an unverified email plus a bearer-style public reference, and that same unverified email is the permanent identity anchor with no correction path. The second cluster is enforcement-point ambiguity: the owner allowlist is specified at page granularity while server actions are independently reachable; the middleware plan conflates /admin with /learn/admin and never covers /api/learn at all; upload validation checks declared metadata while the request_id the paths depend on doesn't exist yet at upload time. None of these require redesign — every fix above is a paragraph-level spec amendment — but all of the high findings should be resolved in the Phase 1 docs before Phase 3 code is written, because each one is the kind of ambiguity an implementer resolves silently in the insecure direction.

[HIGH] (Security — p) PRD.md §6 (journey steps 5–7) vs DATA-MODEL.md §3–4 vs NOTIFICATIONS.md §1 row 7
  ISSUE: The learner-actor transitions quoted→awaiting_payment (accept quote) and awaiting_payment→payment_verification (submit payment evidence) occur BEFORE the learner ever has portal access: PRD journey step 7 and NOTIFICATIONS row 7 both introduce the magic-link portal only AFTER payment is verified. DATA-MODEL §4 defines the Learner role (accept quote, submit evidence) as magic-link-authenticated — an internal contradiction. No document specifies how a pre-portal learner authenticates these state-changing actions. The likely implementation is an unauthenticated endpoint or emailed link keyed by the public reference GNB-LL-xxxx, which is a bearer capability that leaks in every email (including emails sent to mistyped/unverified addresses, row 1). Anyone holding the reference could accept a quote in the learner's name (creating billing disputes: 'I never accepted this scope') or spam awaiting_payment⇄payment_verification transitions, flooding the owner with verify prompts (row 6).
  FIX: Amend PRD §6 and NOTIFICATIONS row 4: portal magic-link sign-in becomes available from `submitted` (or at latest `quoted`), and quote acceptance, clarification replies, and payment-evidence submission are ONLY possible inside an authenticated portal session whose auth.email() matches the request email. Move 'portal magic-link instructions' from row 7 to row 1/4. State explicitly in DATA-MODEL §3 guards: 'learner-actor transitions require an authenticated session bound to the request; the public reference is never an authentication factor.'

[HIGH] (Security — p) DATA-MODEL.md §2 (learning_requests.email 'identity anchor until user_id set') + §4, INTAKE-SCHEMA.md §2 (email 'confirmation NOT required')
  ISSUE: The intake email is never verified, yet it is (a) the identity anchor for the whole engagement, (b) the destination for PII-bearing emails (full name, preliminary assessment, quote, reference — NOTIFICATIONS rows 1,2,4), and (c) the future magic-link account binding. Consequences: a mistyped email permanently locks the real learner out (no email-correction mechanism exists in any doc) and hands whoever controls that inbox full takeover of the engagement — quote, portal, and every uploaded file (academic briefs, personal documents). An attacker can also pre-register requests under a victim's email: the victim's future portal shows attacker-created requests with forged consent records in their name, and each submission fires a legitimate goodnbad.info email at the victim containing attacker-controlled free text (fullName, topic, desiredOutcome) — a phishing/email-bombing vector through your own SMTP that only per-IP limiting does not stop.
  FIX: Add to INTAKE-SCHEMA §5 and DATA-MODEL: (1) double-opt-in — the confirmation email carries only a verify link; the preliminary assessment and reference are shown only after the link is clicked (or after first magic-link sign-in), and screening does not start until verified; (2) per-email rate limit (e.g., max 2 unverified requests per email per 24h) alongside per-IP; (3) an owner-mediated email-correction procedure (owner edits email → all prior magic-link bindings and outstanding references invalidated, event logged); (4) define linking rule for multiple requests per email: bind by user_id after first sign-in, and new email-matching requests attach only after the learner re-verifies via magic link.

[HIGH] (Security — p) PRD.md §8 (admin surface) + DATA-MODEL.md §4 (Owner role)
  ISSUE: PRD §8 gates 'a minimal page at /learn/admin' with the LEARN_OWNER_EMAILS allowlist, with owner server actions 'reachable from' that page. Next.js server actions are independently invokable public HTTP endpoints — gating the page does not gate the actions. As specified, an implementer can ship page-level auth with unauthenticated actions, making payment-confirm and content-release (the two invariants the whole product depends on) callable by anyone. Separately, the allowlist is only as strong as the auth provider config: if the Supabase project leaves email+password or OAuth sign-up enabled (defaults), an attacker may mint a session for an owner email without controlling the inbox, and case/whitespace differences in LEARN_OWNER_EMAILS parsing can silently deny or (worse, with sloppy substring matching) over-grant.
  FIX: Amend PRD §8 and DATA-MODEL §4 hard invariants: (1) 'Every owner server action and route handler begins with requireOwner(): validates the Supabase session server-side, requires the email_verified/confirmed claim, and compares normalized (trimmed, lowercased) auth email for exact membership in LEARN_OWNER_EMAILS — page-level gating is never sufficient.' (2) Add an auth-hardening requirement: Supabase project has ONLY email OTP/magic-link enabled; password and OAuth sign-ups disabled; document this as a launch checklist item alongside removing ADMIN_KEY.

[HIGH] (Security — p) INTAKE-SCHEMA.md §1 ('no server draft persistence in MVP') vs §4 (files upload at Step 4) vs DATA-MODEL.md §2 (bucket_path = requests/{request_id}/…)
  ISSUE: Chicken-and-egg design hole: files are uploaded at Step 4 via signed upload URLs to `requests/{request_id}/…`, but the learning_requests row — and therefore request_id — is only created at Step 6 submit, and §1 forbids server-side draft persistence. Either a draft row must be created at upload time (contradicting §1) or files go to a temp path and are re-bound at submit — and no doc specifies the rebinding. If the client passes file ids/paths at submit, an attacker can bind OTHER people's uploaded files to their own request (IDOR-at-bind: leaks a stranger's academic brief into the attacker's engagement, which the owner then handles as the attacker's). Additionally, signed-URL issuance happens pre-auth for anonymous visitors: without issuance caps and orphan cleanup, a bot can farm URLs and fill the private bucket (storage-cost DoS) with objects no request ever references.
  FIX: Resolve explicitly in INTAKE-SCHEMA §4: server mints an opaque signed intake-session token at Step 4 start; uploads go to `intake/{session_id}/{uuid}.{ext}`; at submit the server binds ONLY files under that session's prefix (client never supplies file ids), moving/relabeling them to `requests/{request_id}/…` server-side. Add: per-IP + per-session caps on URL issuance (e.g., ≤6 URLs/session, ≤20/IP/day), short URL TTL, and a daily cleanup job deleting unbound `intake/*` objects older than 24h.

[MEDIUM] (Security — p) PRD.md §7/§8/§9 (middleware plan) vs middleware.ts:21-25, 47-56
  ISSUE: Three prefix-matching pitfalls in the plan. [MEDIUM] (Security — p) INTAKE-SCHEMA.md §4 + DATA-MODEL.md §2 (learning_files) — MIME/size enforcement point
  ISSUE: 'Allowlist + cap enforced server-side before upload URL issued' validates only client-DECLARED name/MIME/size; the bytes actually PUT to the signed URL can be anything (a 10MB+ binary, text/html, an executable renamed .txt). Nothing in the design pins enforcement at the storage layer or after upload. The allowlist itself contains opaque containers (zip — bombs expand far beyond the 10MB cap; ipynb — JSON that embeds HTML/JS outputs), and the serving spec (attachment-only route) does not mention Content-Type handling, so a naive implementation serving stored MIME could render learner HTML in-origin — amplified by the already-weak CSP (PHASE0-EVIDENCE risk #6).
  FIX: Specify in DATA-MODEL §2: (1) `learner-files` bucket created with Supabase bucket-level `file_size_limit` (10MB) and `allowed_mime_types` matching the allowlist; (2) post-upload server verification before the registry row is marked valid — HEAD the object, compare actual size, sniff magic bytes vs claimed extension, mark mismatches `flagged`; (3) the download route always serves `Content-Type: application/octet-stream` + `Content-Disposition: attachment` + `X-Content-Type-Options: nosniff`, never the stored MIME.

[MEDIUM] (Security — p) NOTIFICATIONS.md §3 (claim-then-send, 'release on hard failure'; 14-day aggregation via POLICIES §3) vs DATA-MODEL.md §2 learning_events ('No UPDATE/DELETE ever issued against this table by application code')

[MEDIUM] (Security — p) POLICIES.md §2 (minors) + §5 (consents.guardian) vs INTAKE-SCHEMA.md §2
